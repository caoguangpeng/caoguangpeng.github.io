OC编码规范

发表于 2015-07-21   |   分类于 iOS   |  

本文参考：

1.Apple Coding Guidelines for Cocoa

2.https://github.com/raywenderlich/objective-c-style-guide

3.程序员头条：http://www.90159.com/2015/06/17/Objective-C编码规范：26个方面解决iOS开发问题/

代码组织

1.方法分组用#pragma mark -来标记；
2.方法组别排序由上至下：

#pragma mark - Lifecycle  
- (void)viewDidLoad {}  
- (void)viewWillAppear:(BOOL)animated {}  
#pragma mark - IBActions  
#pragma mark - Public
#pragma mark - Private
#pragma mark - Protocol conformance
#pragma mark - Custom Accessors(这个在有的编码规范里是放在Lifecycle后面的，考虑到很多变量的初始化可以放在getter方法里面，代码比较长，放到最后比较适合)

缩进

4个空格，可以设置1个Tab=4个空格，在Xcode-Preferences-Text Editing-Indentation里设置

命名

Apple命名规则尽可能坚持，长的，描述性的命名是好的。清晰比简洁更重要,由于 Xcode 强大的自动补全功能，我们不必担心名称过长的问题。
应该：

UIButton *settingsButton;

不应该：

UIButton *setBut;

常量应该使用驼峰式命名规则，所有的单词首字母大写，比如：

static NSTimeInterval const TutorialDuration = 0.3;

变量也是使用驼峰式，但首单词的首字母小写，比如：

descriptiveVariableName

如果BOOL属性的名字是一个形容词，属性就能忽略”is”前缀，但要指定get访问器的惯用名称。例如：

@property (assign, getter=isEditable) BOOL editable;

使用前缀

如果代码需要打包成 Framework 给别的工程使用，或者工程项目非常庞大，需要拆分成不同的模块，使用命名前缀是非常有用的。

前缀由大写的字母缩写组成，一般是2到3个大写之母，比如 Cocoa 中 NS 前缀代表 Founation 框架中的类， IB 则代表 Interface Builder 框架。

可以在为类、协议、函数、常量以及 typedef 宏命名的时候使用前缀，但注意 不要为成员变量或者方法使用前缀，因为他们本身就包含在类的命名空间内。

命名前缀的时候不要和苹果 SDK 框架冲突。

下划线

当使用属性时，实例变量应该使用self.来访问和改变。但有一个特例：在初始化方法里，实例变量(例如，_variableName)应该直接被使用来避免getters/setters潜在的副作用。

局部变量不应该包含下划线。

尽量的使用属性而非实例变量，如果真用实例变量，其命名必须使用下划线 _ 作为前缀，这样可以缩小Xcode自动完成的选项取值范围。
应该：

@interface RNCSection: NSObject
@property (nonatomic) NSString *headline;
@end

不应该：

@interface RNCSection : NSObject {
    NSString *_headline;
}

方法

在方法签名中，应该在方法类型(-/+ 符号)之后有一个空格。在方法各个段之间应该也有一个空格(符合Apple的风格)。在参数之前应该包含一个具有描述性的关键字来描述参数。另外不能用“and”这个词。

应该：

- (void)setExampleText:(NSString *)text image:(UIImage *)image;  
- (void)sendAction:(SEL)aSelector to:(id)anObject forAllCells:(BOOL)flag;

不应该：

- (instancetype)initWith:(int)width and:(int)height;

属性特性

所有属性特性应该显式地列出来，有助于新手阅读代码。属性特性的顺序应该是storage、atomicity，与在Interface Builder连接UI元素时自动生成代码一致。
应该：

@property (weak, nonatomic) IBOutlet UIView *containerView;  
@property (copy, nonatomic) NSString *tutorialName;  (NSString为什么用copy，因为即使你声明一个NSString的属性，有人可能传入一个NSMutableString的实例，然后在你没有注意的情况下修改它。)

不应该：

@property (nonatomic, weak) IBOutlet UIView *containerView;  
@property (nonatomic) NSString *tutorialName;

备注下：定义一个属性时，编译器会自动生成线程安全的存取方法（ Atomic ），但这样会大大降低性能，特别是对于那些需要频繁存取的属性来说，是极大的浪费。所以如果定义的属性不需要线程保护，记得手动添加属性关键字 nonatomic 来取消编译器的优化。

点语法

不要用点分语法来调用方法，只用来访问属性，这样是为了防止代码可读性问题。

应该：

NSInteger arrayCount = [self.array count];  
view.backgroundColor = [UIColor orangeColor];  
[UIApplication sharedApplication].delegate;

不应该：

NSInteger arrayCount = self.array.count;  
[view setBackgroundColor:[UIColor orangeColor]];  
UIApplication.sharedApplication.delegate;

私有属性

私有属性应该在类的实现文件中的类扩展(匿名分类)中声明

@interface RWTDetailViewController ()  
@property (strong, nonatomic) GADBannerView *googleAdView;  
@property (strong, nonatomic) ADBannerView *iAdView;  
@property (strong, nonatomic) UIWebView *adXWebView;  
@end

Init方法

Init方法应该遵循Apple生成代码模板的命名规则，返回类型应该使用instancetype而不是id。

- (instancetype)init {  
  self = [super init];  
  if (self) {  
    // ...  
  }  
  return self;  
}

类构造方法

当类构造方法被使用时，它应该返回类型是instancetype而不是id。这样确保编译器正确地推断结果类型。

@interface Airplane  
+ (instancetype)airplaneWithType:(RWTAirplaneType)type;  
@end

CGRect函数

当访问CGRect里的x, y, width, 或 height时，应该使用CGGeometry函数而不是直接通过结构体来访问。
应该：

CGRect frame = self.view.frame;  
CGFloat x = CGRectGetMinX(frame);  
CGFloat y = CGRectGetMinY(frame);  
CGFloat width = CGRectGetWidth(frame);  
CGFloat height = CGRectGetHeight(frame);

不应该：

CGRect frame = self.view.frame;  
CGFloat x = frame.origin.x;  
CGFloat y = frame.origin.y;  
CGFloat width = frame.size.width;  
CGFloat height = frame.size.height;

黄金路径

当使用条件语句编码时，左手边的代码应该是”golden” 或 “happy”路径。也就是不要嵌套if语句
应该：

- (void)someMethod {  
  if (![someOther boolValue]) {  
    return;  //先把异常情况写出来，突出下面重要的代码
  }  
  //Do something important  
}

不应该：

- (void)someMethod {  
  if ([someOther boolValue]) {  
    //Do something important  
  }  
}
